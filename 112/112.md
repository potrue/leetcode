## 何も見ずに解いてみる

```python
class Solution:
    def hasPathSum(self, root: Optional[TreeNode], targetSum: int) -> bool:
        if not root:
            return False
        stack = [(root, root.val)]
        while stack:
            node, current_sum = stack.pop()
            if not node.left and not node.right:
                if current_sum == targetSum:
                    return True
            if node.left:
                stack.append((node.left, current_sum + node.left.val))
            if node.right:
                stack.append((node.right, current_sum + node.right.val))
        return False
```

## 色々調べてみる

https://github.com/tokuhirat/LeetCode/pull/25/files current_sumのところをループの内部で変えている。sumの値がループの内部で変わってしまうが、親のループで子のsumの面倒を見る必要がないというのはメリット。  
https://github.com/shintaro1993/arai60/pull/29/files targetSumを減らしていく発想はなかった。stackでも今のtargetSum - pathSumを入れていく方式に変えれば実現できそう。ある情報を組にして渡していくDFSは、それを引数に入れた関数で再帰を行うこととほぼ同じか。  
https://github.com/seal-azarashi/leetcode/pull/24/files

この問題のようなどっちでもよさそうなケースでDFSを使うかBFSを使うかはいつもあんまり決めていなかったが、空間計算量を考えるとDFSのほうがよさそう。（logN vs N)

次のnodeがNoneであってもとりあえずstackに入れてしまう実装も考えたが、結局rootがNoneかどうかの条件分岐は消せないことと、単純に今のnodeを渡すだけだとNoneのループから親のもう片方のスロットが空であるかを判別できない（親がleafノードかどうかわからない）のであんまりいい感じじゃなかった。

書いてみる前にこういうのを想像できるようになりたい。というかコードの簡潔さのためにこういう（Noneでもスタックに入れるみたいな）あんまり直感的じゃない実装をすること自体もしかしたら避けるべきなのかな。

class Solution:
    def hasPathSum(self, root: Optional[TreeNode], targetSum: int) -> bool:
        if not root:
            return False
        stack = [(root, root.val)]
        while stack:
            node, current_sum = stack.pop()
            if not node.left and not node.right:
                if current_sum == targetSum:
                    return True
            if node.left:
                stack.append((node.left, current_sum + node.left.val))
            if node.right:
                stack.append((node.right, current_sum + node.right.val))
